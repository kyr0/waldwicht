## Future Research Plan

As part of the Waldwicht project, we are developing `autokernel-mlx`, an open-source system to auto-regressively generate, optimize, evaluate, and compile custom Metal kernels for Apple Silicon inference. The motivation for this effort follows directly from the main result of the present study: weight compression alone is not sufficient. While Waldwicht shows that component-level mixed-precision quantization can preserve answer quality at substantially lower storage cost, the runtime stack still leaves part of these gains unrealized. In particular, the current MLX execution path does not yet fully exploit the structure introduced by heterogeneous bit-width assignments, and therefore cannot convert all compression-time improvements into proportional inference-time speedups or memory reductions. `autokernel-mlx` is intended to close this gap by treating quantization and kernel generation as two halves of the same optimization problem.

The central idea is straightforward. Waldwicht decides *what* precision each component should use in order to preserve model quality under a constrained memory budget. `autokernel-mlx` is designed to decide *how* those precision choices should be executed efficiently on Apple Silicon. Instead of assuming a single generic dequantization path, it searches for specialized kernels that are adapted to concrete quantization layouts, target hardware characteristics, and model execution patterns. In this sense, `autokernel-mlx` extends the logic of Waldwicht from quality-aware quantization into throughput-aware kernel synthesis. The long-term goal is a unified system in which compression policies and inference kernels are co-designed rather than treated as independent stages.

Three concrete research directions motivate this next step.

### 12.1 Dynamic precision at inference time

The first direction is **dynamic precision at inference time**. In the current Waldwicht pipeline, precision is assigned statically at export time. Once a checkpoint has been written, each layer or component remains fixed at its designated bit-width for all future inputs. This is simple and robust, but it is likely suboptimal. Not all tokens, contexts, or decoding steps impose the same reasoning burden on the model. Conversational continuations, boilerplate completions, and structurally predictable spans often tolerate more aggressive quantization than code generation, symbolic reasoning, or mathematically dense prompts. A static assignment therefore forces the model to pay the cost of higher precision even when the input does not require it, or conversely to accept lower precision even when the task would benefit from temporarily increasing it.

A natural extension is a lightweight runtime dispatch mechanism that selects between multiple precompiled kernels for the same component. For example, a layer could have both 3-bit and 4-bit execution paths available, with a controller deciding which one to invoke based on prompt features, token class, or decoder-state heuristics. Under this design, conversational text might remain on the cheaper 3-bit path, while code- or math-heavy segments trigger 4-bit execution for the most sensitive components. The expected advantage is that average inference cost would approach that of a lower-bit configuration, while worst-case quality would remain closer to that of a higher-bit model.

Achieving this efficiently requires more than a policy layer. The key technical challenge is avoiding a precision switch that itself becomes expensive. If moving from 3-bit to 4-bit execution requires reloading weights, rebuilding buffers, or traversing separate host-side code paths, the gain will largely disappear. What is needed instead is a fused runtime design in which multiple quantization formats can coexist behind a single dispatch abstraction, ideally using Metal argument buffers and pre-packed weight representations that allow precision to be selected with minimal overhead. This would effectively introduce *adaptive precision scheduling* at inference time, turning bit-width into a dynamic variable rather than a static export choice.

### 12.2 Fused mixed-precision dequantization kernels

The second direction is **fused mixed-precision dequantization kernels**. One of the clearest lessons from this study is that component-level mixed precision matters: attention, MLP, gating, embeddings, and other structures do not tolerate quantization equally. However, this benefit creates a new runtime burden. A single forward pass through a mixed Waldwicht checkpoint may touch 2-bit, 3-bit, 4-bit, and 5-bit weights in rapid succession. If all of these are funneled through a uniform dequantization path that materializes intermediate BF16 tensors before matrix multiplication, the runtime stack pays a bandwidth and memory penalty precisely where the quantization scheme is trying to save it.

This suggests a more aggressive systems approach: dequantization should be fused directly into the matrix multiplication kernel for each quantization format. Instead of unpacking quantized values into a temporary higher-precision representation and then feeding them into GEMM, the kernel should read packed low-bit blocks, apply the corresponding scaling logic on the fly, and accumulate directly into the compute path. Such kernels would eliminate unnecessary intermediate storage, reduce memory traffic, and better exploit the fact that on Apple Silicon, memory bandwidth and cache behavior are often the real limiting factors during inference.

For Waldwicht specifically, this is particularly important because the benefit of heterogeneous precision depends not only on which bits are assigned where, but also on whether the runtime can execute that heterogeneity without friction. A model that is carefully compressed into a 3/4/5-bit layout but dequantized through a generic one-size-fits-all path leaves performance on the table. In that sense, fused mixed-precision kernels are not merely an optimization; they are a necessary complement to the quantization strategy itself. Future work will therefore explore kernel families specialized by bit-width, group size, scale layout, and component role, with automated benchmarking to identify the best variant for each target chip generation.

### 12.3 KV-cache quantization kernels

The third direction is **KV-cache quantization**. The present work focuses on model weights, but at inference time the KV cache is often a substantial and growing memory cost, especially for long contexts and batch sizes above one. This becomes even more relevant in Gemma 4, where KV-cache sharing creates reuse patterns that make cached key/value tensors disproportionately important for runtime memory efficiency. In the E2B configuration, the last 20 layers reuse earlier caches; in the E4B configuration, 18 layers do so. This means that KV tensors are read repeatedly over the course of decoding, making them an attractive target for compression.

Unlike weight quantization, KV-cache quantization operates on activations produced during inference rather than on static parameters. This introduces a different optimization problem. The system must quantize efficiently on write, store compactly in memory, and dequantize cheaply on read, all while preserving enough fidelity to avoid degrading attention quality across long decoding horizons. A naïve implementation may reduce memory usage but lose the benefit through poor write/read throughput or cumulative numerical drift. The goal of `autokernel-mlx` is to explore custom kernels that handle this lifecycle end to end: quantize-on-write when new cache entries are produced, maintain compact packed storage, and dequantize-on-read only at the point of use.

This direction is especially promising because it is orthogonal to the main Waldwicht quantization scheme. A 4-bit or 8-bit KV cache can be composed with any weight-only configuration described in this paper, offering an additional reduction in runtime memory cost without requiring changes to the quantized checkpoint itself. In practice, this means that even configurations already near the limit of acceptable weight quantization may still gain substantial deployment benefits through cache compression. Future work will therefore investigate the interaction between weight precision, cache precision, sequence length, and cache reuse patterns, with particular emphasis on long-context inference on Apple Silicon systems with constrained unified memory.

### 12.4 LLM-in-the-loop kernel synthesis

To support these directions, `autokernel-mlx` uses an **LLM-in-the-loop generate-evaluate-refine workflow**. Given a kernel specification and a target hardware profile, the system generates candidate Metal Shading Language (MSL) implementations, compiles them, benchmarks them against a reference kernel, and iteratively refines the design based on measured performance. This mirrors the broader methodology used in Waldwicht itself: a search process proposes candidates, objective evaluation filters them, and only the best-performing variants are retained. The difference is that the optimization target shifts from answer quality and memory footprint to runtime throughput, latency, and memory traffic.

This workflow is useful because the Metal kernel design space is large, hardware-sensitive, and often not easily optimized by hand for every combination of bit-width, packing layout, and tensor shape. Tile sizes, threadgroup layouts, packing strategies, memory access patterns, and fusion opportunities interact in ways that are difficult to predict analytically. An iterative synthesis-and-benchmark loop allows the system to search this space empirically. Over time, this could yield a library of specialized kernels matched to the quantization structures that Waldwicht actually produces, rather than relying on generic kernels designed for more uniform quantization schemes.

### 12.5 Toward a unified compression–execution stack

More broadly, this line of work points toward a unified compression–execution stack for local inference on Apple Silicon. In Waldwicht V1, quantization is the primary focus: the contribution is a quality-aware component-level bit allocation strategy and the infrastructure needed to make it work in MLX for Gemma 4. In the next stage, we aim to connect this quantization logic more directly to the runtime layer. That includes not only kernel specialization, but also future extensions such as adaptive precision policies, re-quantization-aware fine-tuning pipelines, fused adapter integration, and runtime policies that jointly optimize quality, latency, and memory consumption.

The long-term vision is therefore not just a better quantizer or a faster kernel generator in isolation, but a system in which the two inform each other. Quantization should produce layouts that are efficient to execute, and execution backends should expose enough structure to make richer quantization policies worthwhile. `autokernel-mlx` is the first planned step in that direction. Its purpose is to turn the theoretical and storage-level gains of Waldwicht into tangible inference-time wins: lower memory pressure, fewer bandwidth bottlenecks, higher throughput, and a deployment stack that makes mixed-precision quantization practical in real-world Apple Silicon inference settings.
